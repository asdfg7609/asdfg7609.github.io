---

"모델을 직접 파인튜닝해보고 싶은데, GPU가 없어요."

LLM에 관심이 생긴 사람들이 가장 먼저 마주치는 벽입니다. 실제로 7B 파라미터 모델을 일반적인 방식으로 파인튜닝하려면 80GB VRAM이 필요하고, 이것은 A100 80GB GPU 한 장 가격, 즉 수천만 원에 해당합니다. Colab 무료 티어가 제공하는 T4 GPU의 VRAM은 16GB입니다. 격차가 너무 큽니다.

이 문제를 해결하기 위해 등장한 것이 LoRA와 QLoRA 같은 파라미터 효율적 파인튜닝 기법이고, Unsloth는 이 기법들을 더 빠르고 더 적은 메모리로 실행할 수 있도록 최적화한 프레임워크입니다.

---

## 1. LLM 파인튜닝의 현실적인 벽

LLM 파인튜닝에 왜 이렇게 많은 VRAM이 필요한지부터 이해해야 합니다.

7B(70억) 파라미터 모델을 예로 들면, 학습 시 GPU가 올려야 하는 데이터는 모델 가중치만이 아닙니다.

```
7B 모델 Full Fine-tuning 시 필요한 VRAM

모델 가중치 (fp16):         14 GB
그래디언트:                 14 GB
옵티마이저 상태 (Adam):     28 GB
활성화 값 (activations):    ~20 GB
──────────────────────────────────
합계:                  약 76~80 GB
```

A100 80GB 한 장으로 겨우 돌아가는 수준입니다. 이것이 "파인튜닝은 돈이 많이 든다"는 인식의 실체입니다.

이 문제를 해결하는 방향은 두 가지입니다. 하나는 **학습하는 파라미터 수를 줄이는 것(LoRA)**이고, 다른 하나는 **가중치를 낮은 정밀도로 저장하는 것(양자화)**입니다. QLoRA는 이 두 가지를 동시에 사용합니다.

---

## 2. LoRA와 QLoRA — 개념 이해

### LoRA: 전체를 바꾸지 말고 차이만 학습하자

**LoRA(Low-Rank Adaptation)**의 핵심 아이디어는 간단합니다. 이미 잘 학습된 모델의 가중치를 전부 바꾸는 대신, **작은 변화량만 학습**하는 것입니다.

비유로 설명하면 이렇습니다. 완성된 그림 위에 얇은 트레이싱지를 덮고, 수정이 필요한 부분만 그 위에 그립니다. 원본 그림(기반 모델)은 그대로이고, 트레이싱지(LoRA 어댑터)에만 변화가 담깁니다.

수식으로 표현하면, 원래 가중치 행렬 $W$를 직접 업데이트하는 대신 저랭크 행렬 두 개의 곱 $\Delta W = BA$를 학습합니다.

$$W' = W + \Delta W = W + BA$$

여기서 $W$는 동결(frozen)되고, $B$와 $A$만 학습됩니다. $r$이 작을수록 학습 파라미터가 적어집니다. 예를 들어 $W$가 4096×4096이라면 약 1680만 개의 파라미터이지만, $r=16$으로 설정한 LoRA 어댑터는 $(4096 \times 16) + (16 \times 4096) = $ 약 13만 개에 불과합니다. **100배 이상 적은 파라미터를 학습**합니다.

```
Full Fine-tuning:
  [기반 모델 가중치 7B] ← 전부 업데이트
  학습 파라미터: 7,000,000,000개

LoRA:
  [기반 모델 가중치 7B] ← 동결(frozen)
  [어댑터 A: 4096×16 ] ← 학습
  [어댑터 B: 16×4096 ] ← 학습
  학습 파라미터: ~131,000개 (약 0.002%)
```

결과적으로 그래디언트와 옵티마이저 상태를 어댑터 파라미터에 대해서만 저장하면 되므로 VRAM이 대폭 줄어듭니다.

### QLoRA: 기반 모델을 4비트로 압축하자

LoRA는 학습 파라미터를 줄였지만, 기반 모델 가중치 자체는 여전히 fp16으로 메모리에 올라가 있습니다. **QLoRA(Quantized LoRA)**는 여기서 한 발 더 나아가, 동결된 기반 모델 가중치를 **4비트 정밀도로 압축**합니다.

4비트 정밀도가 의미하는 것은 이렇습니다. 기존 fp16은 숫자 하나를 16비트(2바이트)로 표현하지만, 4비트는 4비트(0.5바이트)로 표현합니다. 메모리가 4분의 1이 됩니다.

```
기반 모델 메모리 비교 (7B 파라미터 기준):

fp32:   7B × 4byte = 28 GB
fp16:   7B × 2byte = 14 GB
4bit:   7B × 0.5byte = 3.5 GB  ← QLoRA의 기반 모델
```

여기서 중요한 점이 있습니다. QLoRA는 **저장은 4비트로 하되, 계산할 때는 bf16으로 복원**합니다. 4비트로 계산하면 정확도가 크게 떨어지기 때문입니다. 저장의 효율과 계산의 정확도를 동시에 잡는 방식입니다.

4비트 양자화에는 **NF4(NormalFloat4)** 방식이 사용됩니다. LLM 가중치는 정규분포를 따르는 경향이 있는데, NF4는 이 분포에 최적화된 16개의 값으로 가중치를 표현합니다. 무작위로 반올림하는 것보다 정보 손실이 훨씬 적습니다.

```
Full FT vs LoRA vs QLoRA 비교 (7B 모델 기준)

방법            학습 파라미터    VRAM      정확도 손실
Full FT         7B (100%)       ~80 GB    없음
LoRA            ~수백만 (0.1%)  ~30 GB    거의 없음
QLoRA (4bit)    ~수백만 (0.1%)  ~6 GB     미미함
```

## 3. Unsloth란 무엇인가

LoRA와 QLoRA는 파인튜닝을 가능하게 만들었지만, 속도와 메모리 효율을 더 끌어올릴 여지가 남아있었습니다. Unsloth는 이 여지를 파고든 프레임워크입니다.

전 NVIDIA 엔지니어인 Daniel Han Chen이 개발한 Unsloth는 한 가지 목표에 집중합니다.

> **"누구나, 어떤 GPU에서도 LLM을 파인튜닝할 수 있게 한다."**

### 어떻게 빠르게 만드는가: 세 가지 핵심 기법

**① Triton 기반 커스텀 GPU 커널**

PyTorch는 범용적으로 설계된 연산 라이브러리라 모든 상황에 최적화되어 있지 않습니다. Unsloth는 LLM 파인튜닝에 특화된 커널을 OpenAI의 Triton 언어로 직접 작성했습니다. LayerNorm, Attention, RMSNorm, SwiGLU 등 핵심 연산이 모두 수동으로 최적화되어 있습니다.

비유하면, PyTorch가 "다양한 요리를 할 수 있는 일반 주방"이라면 Unsloth는 "파스타만 만드는 데 특화된 주방"입니다. 범용성을 포기한 대신 특정 작업에서 압도적인 효율을 냅니다.

**② Kernel Fusion (연산 융합)**

일반적인 구현에서는 각 연산이 독립적으로 실행됩니다. GPU 메모리에서 데이터를 읽어오고 → 연산하고 → 다시 메모리에 쓰는 과정을 연산마다 반복합니다. 이 메모리 왕복이 병목입니다.

Unsloth는 여러 연산을 하나의 커널로 묶어(fusion) 메모리 왕복을 최소화합니다. 중간 결과를 메모리에 저장하지 않고 레지스터에서 바로 다음 연산으로 전달합니다.

**③ 스마트 Gradient Checkpointing**

역전파(backpropagation) 시 모든 중간 활성화 값(activation)을 메모리에 유지하면 VRAM이 폭발적으로 증가합니다. Gradient Checkpointing은 이것을 저장하지 않고 필요할 때 재계산하는 방식입니다. 속도를 조금 희생하는 대신 메모리를 절약합니다.

Unsloth의 `"unsloth"` 전용 Gradient Checkpointing은 표준 구현보다 약 30% VRAM을 추가로 절약합니다.

### 실측 성능

실제 벤치마크에서 Llama-3.1 8B를 A100 40GB에서 QLoRA로 2 에폭 파인튜닝할 때 Unsloth는 3.2시간 만에 완료했습니다. 표준 구현 대비 약 2배 빠른 수치입니다.

Unsloth는 FA2(Flash Attention 2) 대비 2~5배 빠르고 80% 적은 메모리를 사용하며, 근사화(approximation)나 양자화에 의존하지 않으므로 정확도 손실이 없습니다.

## 4. 전체 파인튜닝 파이프라인 개요

코드를 작성하기 전에 전체 흐름을 먼저 파악합니다. 각 단계가 왜 존재하는지 이해하면 파라미터 선택이 훨씬 명확해집니다.

```
┌─────────────────────────────────────────────────────────────┐
│  Step 1: 환경 설치                                           │
│  unsloth, trl, bitsandbytes 등 의존성 설치                   │
└────────────────────────┬────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────────┐
│  Step 2: 모델 로드 (FastLanguageModel.from_pretrained)       │
│  기반 모델을 4bit 양자화 상태로 메모리에 올림                 │
│  → 기반 모델 가중치 동결                                     │
└────────────────────────┬────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────────┐
│  Step 3: LoRA 어댑터 설정 (FastLanguageModel.get_peft_model) │
│  어느 레이어에 어댑터를 붙일지, rank는 얼마로 할지 설정       │
│  → 학습할 파라미터(어댑터)만 활성화                          │
└────────────────────────┬────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────────┐
│  Step 4: 데이터셋 준비                                       │
│  채팅 템플릿 적용, 토크나이징, 학습용 포맷으로 변환           │
└────────────────────────┬────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────────┐
│  Step 5: 학습 (SFTTrainer)                                   │
│  어댑터 파라미터만 업데이트, 기반 모델은 그대로              │
└────────────────────────┬────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────────┐
│  Step 6: 저장 및 추론                                        │
│  어댑터만 저장 또는 기반 모델에 병합해서 저장                 │
│  선택적으로 GGUF로 변환 → Ollama에서 로컬 실행               │
└─────────────────────────────────────────────────────────────┘
```

## 5. Qwen3.5-4B 4bit LoRA 파인튜닝 — 단계별 코드

### Step 1: 환경 설치

```python
%%capture
# Unsloth와 필수 의존성 설치
# %%capture는 Colab에서 설치 로그를 숨겨주는 매직 커맨드
!pip install unsloth bitsandbytes accelerate peft trl datasets
```

> {: .prompt-info }
> **Colab 환경:** 런타임 → 런타임 유형 변경 → T4 GPU (무료) 또는 A100 (Colab Pro)로 설정하고 실행하세요.

### Step 2: 모델 로드

```python
from unsloth import FastLanguageModel
import torch

# 기본 설정값
MAX_SEQ_LENGTH = 2048   # 처리할 최대 토큰 수 (VRAM과 트레이드오프)
DTYPE = None            # None: GPU에 따라 자동 선택 (Ampere 이상 bfloat16)
LOAD_IN_4BIT = True     # True: QLoRA (4bit), False: LoRA (bf16)

model, tokenizer = FastLanguageModel.from_pretrained(
    model_name    = "unsloth/Qwen3.5-4B-Instruct-bnb-4bit",
    max_seq_length = MAX_SEQ_LENGTH,
    dtype          = DTYPE,
    load_in_4bit   = LOAD_IN_4BIT,
)
```

`FastLanguageModel.from_pretrained`는 Hugging Face의 `AutoModelForCausalLM.from_pretrained`를 Unsloth가 최적화한 래퍼입니다. 내부적으로 모델의 주요 레이어를 Triton 커널로 교체하고, 4bit 양자화를 적용해 로드합니다.

`model_name`에서 `-bnb-4bit`가 붙은 것은 Unsloth 팀이 미리 Dynamic 4bit 양자화를 적용해 Hugging Face에 업로드한 버전입니다. 직접 양자화하는 것보다 정확도가 더 좋습니다.

### Step 3: LoRA 어댑터 설정

```python
model = FastLanguageModel.get_peft_model(
    model,
    r              = 16,           # LoRA 랭크 (rank)
    lora_alpha     = 16,           # 스케일링 계수
    lora_dropout   = 0,            # 드롭아웃 (0이 Unsloth 최적화에 유리)
    bias           = "none",       # 편향 학습 여부
    target_modules = [             # LoRA 어댑터를 붙일 레이어
        "q_proj", "k_proj",        # Attention 레이어
        "v_proj", "o_proj",
        "gate_proj", "up_proj",    # MLP 레이어
        "down_proj",
    ],
    use_gradient_checkpointing = "unsloth",  # Unsloth 전용 최적화
    random_state   = 3407,         # 재현성을 위한 시드
)
```

각 파라미터의 의미와 선택 기준은 다음 섹션(6)에서 상세히 설명합니다.

### Step 4: 데이터셋 준비

파인튜닝에는 원하는 입출력 쌍이 담긴 데이터셋이 필요합니다. 여기서는 공개 데이터셋을 예시로 사용합니다.

```python
from datasets import load_dataset
from unsloth.chat_templates import get_chat_template

# 채팅 템플릿을 토크나이저에 적용
# Qwen3.5는 "qwen-2.5" 템플릿 사용
tokenizer = get_chat_template(tokenizer, chat_template="qwen-2.5")

# 예시 데이터셋 로드 (실제로는 자신의 데이터셋으로 교체)
dataset = load_dataset("philschmid/dolly-15k-oai-style", split="train")

def format_sample(examples):
    """
    데이터셋의 대화를 모델이 이해하는 텍스트 형식으로 변환.
    apply_chat_template이 시스템 프롬프트, 사용자 입력,
    어시스턴트 응답을 올바른 형식으로 조합.
    """
    texts = tokenizer.apply_chat_template(
        examples["messages"],
        tokenize=False,           # 텍스트로만 반환 (토크나이징은 Trainer가)
        add_generation_prompt=False,
    )
    return {"text": texts}

dataset = dataset.map(format_sample, batched=True)
```

> {: .prompt-warning }
> **자신의 데이터셋을 쓸 때 주의:** 데이터가 `messages` 형식 (`[{"role": "user", "content": "..."}, {"role": "assistant", "content": "..."}]`)으로 구성되어야 합니다. 이 형식이 맞지 않으면 채팅 템플릿 적용에서 에러가 납니다.

### Step 5: 학습

```python
from trl import SFTTrainer, SFTConfig

trainer = SFTTrainer(
    model      = model,
    tokenizer  = tokenizer,
    train_dataset = dataset,
    args       = SFTConfig(
        # ── 학습 기본 설정 ───────────────────────────────
        per_device_train_batch_size  = 2,    # GPU 당 배치 크기
        gradient_accumulation_steps  = 4,    # 실효 배치 = 2×4 = 8
        num_train_epochs             = 1,    # 학습 에폭 수
        max_steps                    = -1,   # -1이면 에폭 기준으로 실행

        # ── 학습률 및 스케줄러 ───────────────────────────
        learning_rate                = 2e-4, # LoRA 파인튜닝 권장 학습률
        warmup_ratio                 = 0.05, # 전체 스텝의 5%를 워밍업에 사용
        lr_scheduler_type            = "cosine",  # 코사인 감쇠 스케줄러

        # ── 정밀도 설정 ─────────────────────────────────
        fp16                         = not torch.cuda.is_bf16_supported(),
        bf16                         = torch.cuda.is_bf16_supported(),
        # bf16을 지원하는 GPU(Ampere 이상)면 bf16, 아니면 fp16 자동 선택

        # ── 시퀀스 길이 ─────────────────────────────────
        max_seq_length               = MAX_SEQ_LENGTH,
        dataset_text_field           = "text",  # 데이터셋에서 텍스트 컬럼명

        # ── 로깅 및 저장 ─────────────────────────────────
        logging_steps                = 10,
        output_dir                   = "./outputs",
        save_strategy                = "epoch",

        # ── 최적화 ──────────────────────────────────────
        optim                        = "adamw_8bit",  # 8bit Adam (VRAM 절약)
        weight_decay                 = 0.01,
        seed                         = 3407,
    ),
)

# 학습 시작
trainer_stats = trainer.train()
```

학습이 시작되면 Loss가 점점 줄어드는 것을 확인할 수 있습니다. T4 GPU에서 4B 모델, 1에폭 기준으로 약 1~2시간 소요됩니다.

## 6. 주요 파라미터 이해하기

코드를 그냥 실행하는 것보다 각 파라미터가 무엇을 의미하는지 알면 자신의 상황에 맞게 조정할 수 있습니다.

### LoRA 핵심 파라미터

| 파라미터 | 역할 | 권장 설정 |
| :--- | :--- | :--- |
| **`r` (rank)** | 어댑터 행렬의 랭크. 클수록 표현력 증가, VRAM 증가 | 8~32. 처음에는 16으로 시작 |
| **`lora_alpha`** | 어댑터 출력에 곱하는 스케일링 계수. 실제 학습률 = `lora_alpha/r × lr` | `r`과 같게 설정 (관례) |
| **`lora_dropout`** | 어댑터에 적용할 드롭아웃 비율 | `0` (Unsloth 최적화 커널과 호환) |
| **`bias`** | 편향(bias) 파라미터도 학습할지 여부 | `"none"` (메모리 절약) |

**`r` 선택 가이드:**

```
태스크가 단순하거나 데이터가 적다 (수백~수천 건)  → r=8
일반적인 파인튜닝 (수천~수만 건)                 → r=16  (권장 시작점)
복잡한 태스크, 도메인 적응 (수만 건 이상)          → r=32~64
```

**`target_modules` 선택 가이드:**

```python
# Attention 레이어만 (최소 설정, VRAM 절약)
["q_proj", "v_proj"]

# Attention 전체 (균형)
["q_proj", "k_proj", "v_proj", "o_proj"]

# Attention + MLP 전체 (최대 표현력, Unsloth 권장)
["q_proj", "k_proj", "v_proj", "o_proj",
 "gate_proj", "up_proj", "down_proj"]
```

일반적으로 MLP 레이어까지 포함할수록 성능이 좋아지지만 학습 파라미터와 VRAM이 늘어납니다. 처음에는 모두 포함하는 것이 무난합니다.

### 학습 관련 파라미터

| 파라미터 | 역할 | 권장 설정 |
| :--- | :--- | :--- |
| **`learning_rate`** | 학습률 | LoRA: `1e-4` ~ `3e-4`, Full FT보다 높게 설정 |
| **`per_device_train_batch_size`** | GPU 당 배치 크기 | OOM 나면 줄이기. T4에서는 2~4 |
| **`gradient_accumulation_steps`** | 실효 배치 크기 = 배치 × 이 값 | 물리 배치를 키우지 못할 때 대신 사용 |
| **`max_seq_length`** | 처리할 최대 토큰 수 | 데이터 길이에 맞게 설정. 클수록 VRAM 증가 |
| **`optim`** | 옵티마이저 종류 | `"adamw_8bit"` (VRAM 절약), `"paged_adamw_8bit"` (OOM 방지) |

### gradient_accumulation 이해하기

배치 크기 32로 학습하고 싶은데 GPU VRAM이 허락하지 않을 때 씁니다.

```
목표: 실효 배치 크기 32

방법 1 (불가): batch_size=32 → OOM 에러
방법 2 (가능): batch_size=2, gradient_accumulation_steps=16
               2×16=32, 16번 역전파를 누적해서 한 번 업데이트
```

## 7. 한계와 다음 단계

### Unsloth의 현실적인 한계

**멀티 GPU 미지원**

Unsloth는 단일 GPU에서만 동작합니다. 여러 GPU로 분산 학습이 필요하다면 Axolotl이나 DeepSpeed를 사용해야 합니다.

**Qwen3.5에서 4bit QLoRA 비권장**

Qwen3.5 모델은 커스텀 Mamba Triton 커널을 사용하며, 이로 인해 QLoRA(4bit) 학습은 공식적으로 비권장됩니다. MoE든 dense 모델이든 bf16 LoRA 방식이 권장됩니다. 이 글의 코드 예시에서 `LOAD_IN_4BIT = True`를 사용했지만, Qwen3.5 모델에서 최선의 결과를 원한다면 `load_in_4bit=False, load_in_16bit=True`로 변경하는 것이 좋습니다.

**데이터 품질이 모든 것을 결정**

Unsloth가 학습 속도를 높여주지만, 파인튜닝의 결과는 데이터 품질에 달려 있습니다. 1만 건의 저품질 데이터보다 1천 건의 고품질 데이터가 훨씬 낫습니다.

### 다음 단계: GGUF 변환과 Ollama 배포

저장한 모델을 로컬에서 실행하려면 GGUF 형식으로 변환한 뒤 Ollama에 등록하면 됩니다.

```bash
# Ollama에서 실행
ollama create my-qwen35-4b -f ./Modelfile
ollama run my-qwen35-4b
```

Modelfile 예시:

```

```

## 정리

| 개념 | 핵심 내용 |
| :--- | :--- |
| **Full Fine-tuning** | 모든 파라미터 학습. 높은 성능, 엄청난 VRAM 필요 |
| **LoRA** | 저랭크 어댑터만 학습. VRAM 대폭 절약, 성능 손실 미미 |
| **QLoRA** | 기반 모델 4bit 압축 + LoRA. VRAM 최소화, 무료 GPU에서 가능 |
| **Unsloth** | Triton 커널 + 연산 융합 + 스마트 Gradient Checkpointing으로 2~5배 가속 |
| **`r` (rank)** | LoRA 표현력 조절. 일반적으로 16이 시작점 |
| **`target_modules`** | Attention + MLP 레이어 모두 포함이 권장 |
| **`gradient_accumulation`** | 배치 크기를 물리적으로 늘릴 수 없을 때 사용 |

Unsloth는 "파인튜닝은 고가 GPU가 있어야 한다"는 통념을 바꿉니다. 무료 Colab T4 GPU 하나로 4B 모델을 파인튜닝할 수 있고, 코드 몇 줄만 바꾸면 더 큰 모델도 도전할 수 있습니다. 처음에는 작은 데이터셋과 낮은 rank(r=8)로 시작해서 동작을 확인하고, 점진적으로 설정을 올려가는 방식을 권장합니다.
