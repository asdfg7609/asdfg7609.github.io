---

넷플릭스는 하루에 수백 번 배포를 합니다. 아마존은 평균 **11.6초에 한 번** 배포한다고 알려져 있습니다. 과거 소프트웨어 팀들이 배포를 두려움의 대상으로 여기며 1년에 한두 번만 감행하던 것과 비교하면 상상하기 어려운 수치입니다.

이 차이를 만들어낸 것이 바로 **DevOps**입니다. 단순한 도구의 차이가 아닙니다. 일하는 방식, 팀 구조, 그리고 문화 전체를 바꾼 것입니다.

## 1. 소프트웨어 개발 방식의 진화

DevOps가 왜 등장했는지 이해하려면, 먼저 소프트웨어 개발 방식이 어떻게 변해왔는지 살펴봐야 합니다.

### 워터폴(Waterfall): 순서대로, 한 번에

1980년대부터 주류를 이룬 **워터폴 모델**은 마치 폭포처럼 각 단계가 순서대로 흘러내려 가는 방식입니다.

```
요구사항 분석 → 설계 → 개발 → 테스트 → 배포
```

이 방식의 특징은 각 단계가 완전히 끝난 후에야 다음 단계로 넘어간다는 것입니다. 설계가 완료되어야 개발을 시작하고, 개발이 끝나야 테스트를 시작합니다. 덕분에 **문서화와 계획이 탄탄**하다는 장점이 있지만, 단점도 명확합니다. 배포까지 6개월~1년이 걸리고, 완성 직전에야 사용자 피드백을 받을 수 있습니다. 그 때문에 "만들었더니 사용자가 원하는 게 아니었다"는 비극이 빈번했습니다.

### 애자일(Agile): 짧게, 자주, 반복적으로

2001년, 소프트웨어 개발자 17명이 모여 **애자일 선언문(Agile Manifesto)**을 발표했습니다. 핵심은 간단합니다. 긴 계획보다는 **짧은 사이클(스프린트)로 빠르게 만들고, 사용자 피드백을 받아 개선**하는 것입니다.

```
계획(1~2주) → 개발 → 테스트 → 배포 → 피드백
    ↑___________________________________|
```

스프린트가 짧아지면서 배포 횟수가 급격히 늘었습니다. 한 달에 몇 번, 심지어 매일 배포가 이루어지게 된 것입니다.

그런데 여기서 새로운 문제가 생겼습니다.

## 2. 개발팀과 운영팀의 충돌: "벽 너머로 던지기"

배포가 잦아질수록 개발팀과 운영팀 사이의 마찰이 커졌습니다. 두 팀의 목표가 근본적으로 충돌했기 때문입니다.

```
개발팀(Dev): "기능을 빨리 배포해야 사용자 피드백을 받을 수 있어!"
운영팀(Ops): "자꾸 배포하면 에러가 발생할 확률이 높아져. 안정성이 최우선이야!"
```

배포를 자주 해야 하는 개발팀과, 안정적인 서비스를 유지해야 하는 운영팀은 서로 다른 성과 지표로 평가받았습니다. 개발팀은 "얼마나 빨리 기능을 출시했는가", 운영팀은 "장애 없이 서비스를 유지했는가". 자연스럽게 장애가 나면 서로 손가락질이 시작됩니다.

> *"It's not my code, it's your machines!"*  
> *"It's not my machines, it's your code!"*

이를 업계에서는 **"벽 너머로 던지기(Throw it over the wall)"** 문제라고 부릅니다. 개발팀이 코드를 완성해 운영팀에 던져버리면, 운영팀은 이게 뭔지도 모른 채 배포해야 하는 상황입니다.

## 3. 2009년, 모든 것이 바뀐 발표

2009년, O'Reilly Velocity 컨퍼런스에서 Flickr의 Technical Operations 부문 SVP **John Allspaw**와 Engineering Director **Paul Hammond**가 역사적인 발표를 합니다.

> **"10+ Deploys per Day: Dev and Ops Cooperation at Flickr"**  
> (하루 10번 이상 배포: Flickr의 개발과 운영 협업)

당시 대부분의 회사가 배포를 6주에 한 번, 길게는 1년에 한두 번 하던 시절, "하루에 10번 이상 배포한다"는 발표는 충격 그 자체였습니다.

그들의 핵심 메시지는 단순했습니다. **개발팀과 운영팀이 같은 목표를 가지고 협업**하면, 배포를 두려워하지 않을 수 있다는 것. "Ops who think like devs, Devs who think like ops"가 그들의 슬로건이었습니다.

이 발표를 원격으로 시청하던 벨기에의 시스템 관리자 **Patrick Debois**는 자신만의 컨퍼런스를 열기로 결심합니다. 같은 해 10월 30일, 벨기에 헨트(Ghent)에서 열린 그 컨퍼런스의 이름이 바로 **DevOpsDays**였고, 트위터 해시태그를 만들면서 이름을 줄여 **#DevOps**가 탄생했습니다.

## 4. 기존 조직의 4가지 핵심 문제

DevOps가 해결하려는 문제들을 구체적으로 살펴봅니다.

### 문제 1: 인프라팀의 무관심 (Silos)

> "그건 저희 일이 아닌데요. 저희는 서버만 관리해요."

조직이 역할별로 분리되면서 **사일로(Silo)** 현상이 발생합니다. 인프라팀은 하드웨어와 네트워크만 담당하고, 애플리케이션의 동작 방식에는 관심이 없습니다. 반대로 개발팀은 코드를 완성하면 "그다음은 운영팀 일"이라고 생각합니다.

이 단절은 배포 당일이 되어서야 수면 위로 드러납니다. 개발 환경에서 잘 작동하던 코드가 운영 환경에서 갑자기 실패하는 이른바 **"내 컴퓨터에서는 잘 됐는데" 문제**가 반복됩니다.

### 문제 2: 운영팀의 기술적 부담과 두려움

운영팀은 실제로 이런 딜레마에 빠집니다.

- 보안 패치를 적용하면 서비스가 죽을 수 있어 적용하지 못한다
- 하드웨어 지원 기간이 끝났지만 교체하기 무서워 그냥 운영한다
- 영향도를 알 수 없으니 어떤 변경도 두렵다

이 상태를 "**영향도를 모르니 움직이기 두려운(Fear of Unknown Impact)**" 상태라고 합니다. 운영팀이 소극적이 되는 건 의지의 문제가 아니라, 시스템의 복잡도와 가시성 부재 때문입니다.

### 문제 3: 스페셜리스트만 있는 팀 구조

> "저는 백엔드 개발자라 프론트엔드는 몰라요."  
> "저는 DBA라 서버 설정은 몰라요."

각 분야의 전문가는 많지만, **전체 그림을 보는 사람**이 없습니다. 배포 과정에서 필요한 요건이 중간에 빠지거나, 각 팀이 자신의 파이프라인 구간만 챙기다 결국 "아무도 전체를 책임지지 않는" 상황이 발생합니다.

### 문제 4: 비기능 요건(Non-functional Requirements)의 방치

개발자들은 기능(Feature)을 만드는 데 집중하지만, 운영의 관점에서 필수적인 항목들을 소홀히 하는 경향이 있습니다.

| 비기능 요건 | 방치 시 결과 |
| :--- | :--- |
| **모니터링** | 서비스가 죽었는데 수십 분 후에 알게 됨 |
| **이중화(HA)** | 서버 하나 장애로 전체 서비스 다운 |
| **백업** | 데이터베이스 손상 시 복구 불가 |
| **보안 업데이트** | 알려진 취약점이 방치됨 |
| **로깅** | 장애 원인을 소급해서 파악 불가 |

## 5. DevOps의 해결책: 문화 + 방법론 + 도구

DevOps를 "CI/CD 도구를 쓰는 것"으로 이해하는 것은 절반의 오해입니다. 도구는 수단일 뿐, 핵심은 **일하는 방식과 문화의 변화**입니다.

```
DevOps = 일하는 방법 + 팀 빌딩 + 개발 프로세스 설계 + 문화의 형성
```

그 철학을 한 문장으로 요약하면:

> **"Dev and Ops see the same thing, in the same place."**  
> 개발팀과 운영팀이 동일한 정보를, 동일한 공간에서 함께 본다.

이를 실현하는 핵심 실천 방법들:

| 실천 방법 | 설명 |
| :--- | :--- |
| **IaC** (Infrastructure as Code) | 인프라 구성을 코드로 관리. 개발자도 인프라를 읽고 이해할 수 있다 |
| **버전 관리 공유** | 소프트웨어 코드와 인프라 설정 모두를 Git으로 통합 관리 |
| **원스톱 배포** | 버튼 하나 또는 PR 머지만으로 빌드 → 테스트 → 배포 완료 |
| **통합 모니터링** | 개발팀·운영팀 모두 동일한 대시보드로 서비스 상태를 확인 |
| **공동 온콜(On-call)** | 장애 발생 시 개발자도 대응에 참여. "내 코드가 터지면 나도 깨어난다" |

## 6. CI란 무엇인가? — 지속적 통합(Continuous Integration)

### 통합을 미루면 생기는 일

팀원 3명이 각자 1주일씩 기능을 개발한 후 코드를 합치려고 했더니...

```
팀원 A: user.js의 getUserInfo() 함수 수정
팀원 B: user.js의 getUserInfo() 함수 수정 (동일 파일!)
팀원 C: A의 함수를 호출하는 코드 작성
        → 근데 A가 내부적으로 함수 시그니처를 바꿨다
```

결과는 예상할 수 있습니다.

- **병합 충돌(Merge Conflict)**: 같은 파일을 서로 다르게 수정
- **빌드 오류**: 변경된 함수 시그니처를 호출하며 런타임 에러
- **잠재적 버그**: 1주일치 변경이 한꺼번에 합쳐지면서 원인 추적 불가

통합을 미룰수록 문제는 기하급수적으로 커집니다. 이것을 **통합 지옥(Integration Hell)**이라고 부릅니다.

### CI의 철학: "아프면, 더 자주 해라 (If it hurts, do it more often)"

**CI(Continuous Integration, 지속적 통합)**의 핵심 아이디어는 단순합니다. 코드가 바뀔 때마다 **자동으로 통합·빌드·테스트**를 수행하면, 문제를 즉시 발견하고 작은 단위로 해결할 수 있다는 것입니다.

```
코드 변경(커밋/PR) → 자동 빌드 → 자동 테스트 → 결과 알림
```

이것이 가능하려면 **VCS(Version Control System)**, 즉 Git이 팀 전체의 단일 진실 공급원(Single Source of Truth)으로 자리 잡아야 합니다.

### CI 파이프라인의 각 단계

| 단계 | 수행 내용 | 실패 시 조치 |
| :--- | :--- | :--- |
| **커밋(Commit)** | 코드를 공유 저장소에 반영 | 충돌 발생 시 개발자에게 즉시 알림 |
| **빌드(Build)** | 코드를 컴파일 또는 번들링 | 빌드 오류를 커밋 직후 발견 |
| **단위 테스트** | 개별 함수·모듈 단위 자동 검증 | 로직 버그를 코드 작성 직후 발견 |
| **통합 테스트** | 여러 모듈 간의 상호작용 검증 | 인터페이스 불일치 조기 발견 |
| **정적 분석** | 코드 스타일, 보안 취약점 패턴 검사 | 코드 품질 기준 미달 시 머지 차단 |
| **E2E 테스트** | 실제 사용자 시나리오 자동 실행 | 전체 흐름 검증 |

실패 시 Slack, 이메일, SMS 등을 통해 담당자에게 즉시 알림을 발송합니다. **문제를 발견하는 시점이 빠를수록 수정 비용은 기하급수적으로 줄어듭니다.**

### 실전 코드: GitHub Actions로 CI 파이프라인 구성

아래는 Python 프로젝트에 CI를 적용하는 GitHub Actions 예시입니다.

```yaml
# .github/workflows/ci.yml

name: CI Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  ci:
    runs-on: ubuntu-latest

    steps:
      # 1. 코드 체크아웃
      - name: Checkout code
        uses: actions/checkout@v4

      # 2. Python 환경 설정
      - name: Set up Python 3.11
        uses: actions/setup-python@v5
        with:
          python-version: "3.11"

      # 3. 의존성 캐싱 (빌드 속도 향상)
      - name: Cache pip dependencies
        uses: actions/cache@v4
        with:
          path: ~/.cache/pip
          key: ${{ runner.os }}-pip-${{ hashFiles('requirements.txt') }}

      # 4. 의존성 설치
      - name: Install dependencies
        run: pip install -r requirements.txt

      # 5. 정적 분석 (코드 스타일 + 타입 체크)
      - name: Lint with ruff
        run: ruff check .

      - name: Type check with mypy
        run: mypy src/

      # 6. 단위 테스트 + 커버리지 측정
      - name: Run tests with coverage
        run: |
          pytest tests/ \
            --cov=src \
            --cov-report=xml \
            --cov-fail-under=80  # 커버리지 80% 미만 시 파이프라인 실패

      # 7. 커버리지 리포트 업로드 (선택)
      - name: Upload coverage to Codecov
        uses: codecov/codecov-action@v4
        with:
          file: ./coverage.xml
```

이 파이프라인은 `main` 브랜치에 PR이 열리거나 코드가 푸시될 때마다 자동으로 실행됩니다. 테스트가 하나라도 실패하면 PR 머지가 차단됩니다.

## 7. CD란 무엇인가? — 지속적 배포(Continuous Delivery/Deployment)

CI가 코드의 품질을 자동으로 검증한다면, **CD**는 검증된 코드를 자동으로 배포 환경에 반영합니다.

### CD의 두 가지 의미

CD에는 미묘하게 다른 두 가지 개념이 있습니다.

| 개념 | 의미 | 인간 개입 여부 |
| :--- | :--- | :--- |
| **Continuous Delivery** (지속적 제공) | 언제든 배포 가능한 상태를 유지. 실제 배포는 사람이 버튼을 누름 | 최종 배포 시 수동 승인 |
| **Continuous Deployment** (지속적 배포) | CI 통과 즉시 프로덕션까지 자동 배포 | 전 과정 자동화 |

초기 단계의 팀은 Continuous Delivery로 시작해 신뢰가 쌓이면 Continuous Deployment로 나아가는 것이 일반적입니다.

### 배포 환경의 단계: Dev → Staging → Production

```
[개발 환경(Dev)]
  개발자 로컬 또는 개발 서버
  - 자유롭게 변경, 깨져도 무방
  ↓ CI 통과 시 자동 배포
[스테이징 환경(Staging)]
  운영과 동일한 구성의 검증 서버
  - 운영 배포 전 최종 통합 테스트
  - QA 팀 수동 테스트 수행
  ↓ 승인 또는 자동 배포
[프로덕션 환경(Production)]
  실제 사용자가 사용하는 운영 서버
  - 장애 발생 시 즉시 롤백
```

스테이징 환경이 중요한 이유는, 개발 환경에서 잘 동작하던 코드가 운영 환경에서 갑자기 문제를 일으키는 경우가 빈번하기 때문입니다. 데이터베이스 버전, 환경 변수, 메모리 설정 등 수많은 차이가 존재합니다.

### 주요 배포 전략

안정적인 CD를 위해 다양한 배포 전략이 활용됩니다.

| 전략 | 방식 | 장점 | 단점 |
| :--- | :--- | :--- | :--- |
| **롤링(Rolling)** | 서버를 순차적으로 교체 | 다운타임 없음 | 일시적으로 구버전·신버전 공존 |
| **블루/그린(Blue/Green)** | 동일 환경 2벌 유지, 트래픽 전환 | 즉각 롤백 가능 | 인프라 비용 2배 |
| **카나리(Canary)** | 일부 트래픽(예: 5%)만 신버전으로 | 위험 최소화, 실제 트래픽 검증 | 모니터링 복잡도 증가 |
| **피처 플래그(Feature Flag)** | 코드는 배포, 기능은 설정으로 ON/OFF | 배포와 릴리즈 분리 | 플래그 관리 오버헤드 |

### CI/CD 전체 파이프라인 흐름

```
[개발]
  이슈 생성 (Jira, GitHub Issues)
       ↓
  브랜치 생성 + 코드 작성
       ↓
  PR(Pull Request) 오픈
       ↓
[CI — 자동 실행]
  빌드 → 단위 테스트 → 통합 테스트 → 정적 분석
       ↓ 실패 → 개발자에게 알림, PR 머지 차단
       ↓ 성공
  코드 리뷰 + 승인
       ↓
  main 브랜치 머지
       ↓
[CD — 자동 실행]
  스테이징 자동 배포
       ↓
  스테이징 검증 (자동화 테스트 + QA)
       ↓ 실패 → 개발자에게 알림, 롤백
       ↓ 성공
  프로덕션 배포 (카나리 → 롤링 또는 직접 배포)
       ↓
[운영]
  모니터링 + 알림 (Prometheus, Grafana, Datadog)
       ↓ 이상 감지
  자동 또는 수동 롤백
```

## 8. DevOps의 성과를 측정하는 방법: DORA 메트릭

DevOps가 잘 되고 있는지를 어떻게 측정할 수 있을까요? 2015년 Dr. Nicole Forsgren, Gene Kim, Jez Humble이 설립한 **DORA(DevOps Research and Assessment)**는 4가지 핵심 지표를 제안했습니다.

| 지표 | 설명 | 엘리트 팀 기준 |
| :--- | :--- | :--- |
| **배포 빈도(Deployment Frequency)** | 얼마나 자주 배포하는가 | 하루 여러 번 |
| **변경 리드 타임(Lead Time for Changes)** | 커밋부터 프로덕션 배포까지 걸리는 시간 | 1시간 미만 |
| **변경 실패율(Change Failure Rate)** | 배포 후 롤백이 필요한 비율 | 5% 미만 |
| **서비스 복구 시간(MTTR)** | 장애 발생 후 복구까지 걸리는 시간 | 1시간 미만 |

Google의 DORA 연구에 따르면, 엘리트 팀은 저성과 팀 대비 **127배 빠른 리드 타임**, **182배 낮은 변경 실패율**, **8배 높은 배포 빈도**, 그리고 **2,293배 빠른 장애 복구**를 기록했습니다. 이 숫자는 단순한 도구의 차이가 아닌, 문화와 프로세스 전체의 차이에서 비롯됩니다.

2026년 기준으로 최상위 조직들은 평균 리드 타임을 **15분** 수준까지 단축하고 있으며, 이는 자동화된 테스트, IaC, 클라우드 네이티브 배포 전략의 조합으로 달성됩니다.

## 9. 정리

| 개념 | 핵심 내용 |
| :--- | :--- |
| **DevOps** | 개발·운영의 협업 문화. 도구보다 마인드셋이 먼저 |
| **CI** | 코드 변경 시 자동 빌드·테스트로 문제를 즉시 발견 |
| **CD** | CI 통과 후 스테이징 → 프로덕션 배포를 자동화 |
| **DORA 메트릭** | 배포 빈도·리드 타임·실패율·복구 시간으로 팀 수준을 측정 |

DevOps의 본질은 기술이 아닙니다. **"배포가 무섭지 않은 팀"**을 만드는 것입니다. 배포가 작아지고 자주 일어날수록, 각 배포의 위험도는 낮아집니다. 롤백이 쉬울수록, 팀은 과감하게 실험할 수 있습니다.
