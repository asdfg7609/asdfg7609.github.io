> **요약:** AI 에이전트 분야는 같은 단어가 맥락마다 다른 뜻으로 쓰이는 용어 혼란이 심합니다. HuggingFace가 2026년 5월 발행한 Agent Glossary를 바탕으로, 실무에서 자주 마주치는 핵심 용어 12개를 정확하게 정리합니다.

---

AI 에이전트를 다루는 글을 읽다 보면 같은 단어가 전혀 다른 의미로 쓰이는 경우가 많습니다. "harness"는 어떤 글에서는 전체 에이전트 시스템을 가리키고, 다른 글에서는 실행 루프만을 지칭합니다. "scaffold"는 프롬프트 구조를 뜻하기도 하고, 코드 인프라 전체를 가리키기도 합니다.

ICLR 2026에서 연구자 Aritra Roy Gosthipaty가 이 혼란을 직접 경험한 뒤 HuggingFace 블로그에 올린 질문이 이 글의 출발점이 됐습니다.

> *"에이전트 맥락에서 'harness'와 'scaffold'가 무슨 뜻인지 설명을 많이 들었지만, 왜 설명이 하나로 수렴하지 않는지 이해할 수가 없었다."*

이 글은 HuggingFace의 Agent Glossary(2026.05.25)를 바탕으로, 실무에서 반복적으로 등장하는 핵심 용어들을 정리합니다. 보편적으로 합의된 정의가 아직 없는 용어도 있습니다. 목표는 정확한 사전 편찬이 아니라, 글을 읽고 대화를 나눌 때 혼동을 줄이는 실용적인 멘탈 모델을 제공하는 것입니다.

## 1. Model

**모델은 LLM 그 자체입니다.** 텍스트를 받아 텍스트를 출력하는 함수입니다. GPT, Claude, Qwen, Gemini 등이 여기에 해당합니다.

모델 단독으로는 에이전트가 아닙니다. 모델은 호출 사이에 메모리가 없고, 루프도 없습니다. 모델은 툴 호출 의도를 텍스트로 표현할 수 있지만, 실제로 그 툴을 실행하려면 하네스가 필요합니다. 프롬프트 하나를 받고 응답 하나를 내보내면 멈춥니다.

모델을 스캐폴딩과 하네스로 감싸면 비로소 에이전트가 됩니다.

---

## 2. Scaffolding

**스캐폴딩은 모델이 세상을 바라보고 행동하는 방식을 정의하는 레이어입니다.** 시스템 프롬프트, 툴 설명, 모델 응답의 파싱 방식, 단계 간 무엇을 기억할지(컨텍스트 관리)가 여기에 해당합니다.

스캐폴딩은 모델의 행동을 형성합니다. 훈련 시에도, 추론 시에도 동일하게 적용됩니다.

혼란이 생기는 이유는 Claude Code, Codex 같은 제품들이 "harness"라는 단어로 이 전체를 통칭하기 때문입니다. Claude Code의 공식 문서는 직접 이렇게 표현합니다: "Claude Code serves as the agentic harness around Claude." 이 맥락에서 harness는 모델이 아닌 모든 것을 의미합니다.

scaffold/harness의 구분이 중요해지는 것은 두 개념을 **분리해서 추론해야 할 때**, 특히 훈련 파이프라인 설계 시입니다. "scaffold"가 하네스가 의존하는 모든 인프라—훅, 런타임 설정, 디렉터리 구조—를 포괄하는 더 넓은 의미로 쓰이는 경우도 있습니다.

---

## 3. Harness

**하네스는 에이전트를 실제로 실행시키는 계층입니다.** 모델을 호출하고, 툴 호출을 처리하고, 언제 멈출지 결정합니다.

스캐폴딩이 모델이 작동하는 기반(지시사항, 툴, 형식)이라면, 하네스는 그 기반 위에서 실제로 루프를 돌리는 실행 엔진입니다.

**하네스 엔지니어링(Harness Engineering)** 은 이 레이어를 잘 설계하는 기술 분야입니다. 에이전트가 언제 멈춰야 하는지, 오류를 어떻게 처리할지, 어떤 가드레일로 통제할지를 결정합니다. 훈련 시에도, 추론 시에도 적용됩니다.

평가(evaluation) 맥락에서는 같은 패턴이 **eval harness**로 나타납니다. 훈련 데이터를 수집하는 대신, 특정 모델 체크포인트에서 고정된 시나리오 집합을 실행하고 가중치를 업데이트하는 대신 지표를 기록합니다.

여러 에이전트를 조율하는 상위 컨트롤러는 **오케스트레이터(Orchestrator)** 라고 부릅니다. 하네스가 단일 모델의 실행 루프를 구동하는 것과 달리, 오케스트레이터는 각자의 하네스를 가진 에이전트들을 단위로 관리합니다.

---

## 4. Agent

강화학습에서 에이전트는 관찰(observation)을 받아 행동(action)을 반환하는 함수입니다. 환경은 그 행동을 받아 상태를 업데이트하고 새로운 관찰을 반환합니다. 이 루프가 반복됩니다. LLM 에이전트도 이 루프를 핵심으로 합니다.

LLM 세계에서 에이전트의 의미는 확장됐습니다. **에이전트는 모델에 그것이 루프 안에서 행동할 수 있도록 하는 모든 것을 더한 시스템입니다.** 정보를 받고, 무엇을 할지 결정하고, 결과에 따라 행동하는, 단순한 텍스트 생성을 넘어서는 것입니다.

커뮤니티에서는 이를 간결하게 표현합니다:

> **Agent = Model + Harness**

모델이 아닌 모든 것이 하네스입니다.

코딩 에이전트를 예시로 구체화하면: 시스템 프롬프트, 툴 설명, 모델이 따르는 출력 형식이 스캐폴딩입니다. 모델을 호출하고 툴 호출을 처리하고 멈출 시점을 결정하는 루프가 하네스입니다. 훈련 시에는 하네스가 이 루프를 병렬로 여러 번 실행하고 결과를 모델 업데이트에 반영합니다.

```
┌─────────────────────────────────────────┐
│                  Agent                  │
│  ┌─────────────────────────────────┐    │
│  │            Harness              │    │
│  │  ┌───────────────────────────┐  │    │
│  │  │        Scaffolding        │  │    │
│  │  │  ┌─────────────────────┐  │  │    │
│  │  │  │        Model        │  │  │    │
│  │  │  └─────────────────────┘  │  │    │
│  │  └───────────────────────────┘  │    │
│  └─────────────────────────────────┘    │
│              Sub-agent(s)               │
└─────────────────────────────────────────┘
```

Claude Code, Codex, Cursor 같은 제품들은 특정 모델 위에 함께 설계·최적화된 특정 하네스입니다. 동일한 기반 모델을 쓰는 두 제품이 완전히 다른 경험을 줄 수 있는 이유가 하네스의 차이 때문입니다. 모델과 하네스와 제품은 세 가지 다른 것입니다.

## 5. Context Engineering

**컨텍스트 엔지니어링은 에이전트의 컨텍스트 창에 무엇을 넣을지 설계하는 것입니다.** 각 단계에서 모델이 보는 것—시스템 프롬프트, 툴 설명, 대화 히스토리, 검색된 지식—을 결정합니다.

일회성 결정이 아닙니다. 모델이 실행되는 동안 이전 턴이 미래 호출에 들어가는 내용을 형성하고, 하네스가 실행 전반에 걸쳐 이를 능동적으로 관리합니다.

훈련과 추론에서 모두 적용되지만, 잘못됐을 때의 비용이 크게 다릅니다.

- **훈련 시:** 모델이 보는 것이 학습되는 것을 형성합니다. 잘못되면 재훈련이 필요합니다.
- **추론 시:** 텍스트일 뿐입니다. 프롬프트를 바꾸고 재배포하면 됩니다.

메모리는 컨텍스트 엔지니어링의 일부입니다.

| 구분 | 설명 |
|:---|:---|
| **단기 메모리 (Short-term)** | 단일 실행 중 컨텍스트 창에 남아있는 것. 대화 히스토리, 툴 결과, 이전 추론 |
| **장기 메모리 (Long-term)** | 세션을 넘어 외부에 저장되고, 필요할 때 검색해 컨텍스트에 주입 |

---

## 6. Policy

**폴리시(Policy)는 에이전트가 따르는 행동 방식입니다.** 어떤 상황에서든 각 행동을 취할 확률 분포를 정의합니다.

LLM 시스템에서 폴리시의 일부는 모델 가중치에 학습되어 있지만, 행동은 주변 스캐폴딩과 하네스에도 의존합니다. **동일한 모델이 프롬프트, 툴, 메모리, 실행 루프에 따라 전혀 다르게 행동할 수 있습니다.**

폴리시는 에이전트가 아닙니다. 폴리시는 행동을 정의하고, 에이전트는 환경 안에서 실제로 행동하는 전체 시스템입니다. 체크포인트를 스캐폴딩과 하네스로 감싸 배포하면, 그 행동이 폴리시인 에이전트가 됩니다.

---

## 7. Tool Use

**툴 사용은 에이전트가 자신의 외부에 닿는 방법입니다.** API, 코드 인터프리터, 데이터베이스, 웹 검색, 파일 시스템이 여기에 해당합니다.

모델은 구조화된 형식으로 툴 사용 의도를 표현합니다. 현대 추론 API는 이를 1등급 객체(first-class object)로 제공합니다. 하네스가 호출을 직접 받아 적절한 함수로 라우팅하고, 결과를 컨텍스트에 다시 주입하면 루프가 이어집니다.

---

## 8. Skills

**스킬은 멀티스텝 작업을 가능하게 하는 재사용 가능한 구조화 패키지입니다.**

툴이 단일 행동("이 명령어를 실행하라")이라면, 스킬은 목표를 달성하는 데 필요한 모든 것을 묶습니다("이 버그를 조사하고, 가설을 세우고, 수정사항을 작성하라"). 에이전트 간에 이식 가능하며, 필요할 때 로드됩니다.

툴, 스킬, 서브에이전트의 경계는 프레임워크마다 다릅니다. 아래 표가 실용적인 구분 기준입니다.

| 개념 | 성격 | 예시 |
|:---|:---|:---|
| **Tool** | 단일 행동, 함수 호출 | 웹 검색, 파일 읽기, 코드 실행 |
| **Skill** | 패키지화된 절차·지식, 멀티스텝 | PR 리뷰 체크리스트, 보고서 작성 SOP |
| **Sub-agent** | 독립적으로 추론하는 에이전트 | 보안 감사 에이전트, 테스트 작성 에이전트 |

---

## 9. Sub-agents

**서브에이전트는 특정 하위 작업을 처리하기 위해 다른 에이전트에 의해 호출되는 에이전트입니다.** 자체 모델과 스캐폴드를 가지며, 독립적으로 추론하고 결과를 반환합니다. 호출한 에이전트는 내부 동작 방식을 알 필요가 없습니다.

이것이 서브에이전트를 툴(함수 호출)이나 스킬(패키지화된 지식)과 구분하는 핵심입니다. **서브에이전트는 스스로 추론하고, 툴을 사용하고, 추가 서브에이전트를 호출할 수 있습니다.** 호출하는 에이전트는 종종 **오케스트레이터**라고 부릅니다.

## 10. RL Environment

이하 네 개 용어는 에이전트 훈련에 특화된 개념입니다. 에이전트가 태스크를 실행하고, 점수를 받고, 모델 가중치가 업데이트되는 과정에서 쓰입니다.

**RL 환경은 에이전트가 상호작용하는 대상입니다.** 행동을 입력으로 받아 내부 상태를 업데이트하고 관찰을 반환하는 상태 있는(stateful) 객체입니다. LLM 맥락에서 행동은 보통 툴 호출입니다.

파일 시스템이 간단한 예시입니다. `touch foo.txt` 행동은 파일을 생성해 상태를 업데이트하고, 업데이트된 파일 목록이 관찰로 반환됩니다.

## 11. Rollout

**롤아웃은 처음부터 끝까지 에이전트가 한 번 실행된 전체 기록입니다.** 에이전트가 무엇을 봤는지, 무엇을 했는지, 각 단계에서 어떤 보상을 받았는지가 담겨 있습니다.

**트레이젝토리(trajectory)** 또는 **트레이스(trace)** 라고도 불립니다. RL 알고리즘이 학습하는 원재료입니다.

```
Rollout 예시:
  step 1 → 관찰: "버그 수정 요청"  행동: read_file("app.py")
  step 2 → 관찰: [파일 내용]       행동: edit_file(line=42, ...)
  step 3 → 관찰: "수정 완료"       행동: run_tests()
  step 4 → 관찰: "테스트 통과"     종료
  reward: +1.0
```

## 12. Reward

**보상(Reward)은 모델이 나아지고 있는지 훈련 알고리즘에 알려주는 점수입니다.**

| 구분 | 설명 | 예시 |
|:---|:---|:---|
| **Verifiable** | 자동으로 검증 가능 | 테스트 통과/실패, 정답 일치 |
| **Learned** | 학습된 선호도 | 인간 피드백, LLM-as-judge |
| **Sparse** | 에피소드 끝에 한 번 | 최종 점수 |
| **Dense** | 각 단계마다 | 단계별 점수 |

**루브릭(Rubric)** 은 단일 숫자 대신 보상을 명시적인 차원과 가중치로 분해합니다. OpenEnv와 Verifiers 같은 프레임워크는 루브릭을 `WeightedSum`, `Sequential`, `Gate` 같은 조합 가능한 객체로 구현합니다.

## 정리: 한눈에 보는 12개 용어

| 용어 | 한 줄 정의 | 적용 시점 |
|:---|:---|:---|
| **Model** | 텍스트 입출력 함수 (LLM) | 항상 |
| **Scaffolding** | 모델이 세상을 보는 방식 정의 (프롬프트·툴 설명·파싱) | 훈련 + 추론 |
| **Harness** | 에이전트 실행 엔진 (루프·툴 실행·종료 결정) | 훈련 + 추론 |
| **Agent** | Model + Harness | 훈련 + 추론 |
| **Context Engineering** | 컨텍스트 창에 무엇을 넣을지 설계 | 훈련 + 추론 |
| **Policy** | 에이전트의 행동 방식 (확률 분포) | 훈련 + 추론 |
| **Tool Use** | 외부 시스템과의 상호작용 | 추론 |
| **Skill** | 재사용 가능한 멀티스텝 지식 패키지 | 추론 |
| **Sub-agent** | 독립 추론하는 하위 에이전트 | 추론 |
| **RL Environment** | 행동을 받아 관찰을 반환하는 상태 객체 | 훈련 |
| **Rollout** | 에이전트 한 번 실행의 전체 기록 | 훈련 |
| **Reward** | 모델 개선 방향을 알리는 점수 | 훈련 |

## 마치며

용어는 도구입니다. 정확한 언어가 없으면 논의가 어긋나고, 시스템 설계가 흐릿해집니다. 이 분야의 용어는 아직 완전히 수렴하지 않았고, 프레임워크마다 같은 단어를 다르게 쓰는 경우도 있습니다. 그래도 공통된 멘탈 모델을 하나 갖고 있으면, 새로운 글과 코드를 읽을 때 기준점이 생깁니다.

글을 쓸 때마다 이 용어들이 혼용되는 것을 느낀다면, 그 자체가 이 분야가 얼마나 빠르게 움직이고 있는지를 보여주는 신호이기도 합니다.
