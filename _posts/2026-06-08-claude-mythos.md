> **요약:** Claude Mythos Preview는 OpenBSD에 27년간 숨어있던 버그를 $50 미만의 비용으로 자율 발견했습니다. Anthropic Frontier Red Team의 공식 기술 문서를 바탕으로, Mythos가 어떤 스캐폴드 구조로 코드를 탐색하고 어떻게 익스플로잇을 구성하는지 해부합니다.

---

버그가 코드베이스에 들어간 시점은 1998년이었습니다.

27년 동안 이 코드는 수백 번의 보안 감사를 통과했습니다. OpenBSD는 보안에 가장 엄격하기로 알려진 운영체제입니다. 코드를 들여다본 전문가들은 수도 없이 많았습니다. 자동화 퍼저도 수없이 돌아갔습니다. 아무도 이 버그를 찾지 못했습니다.

이 버그를 발견한 것은 Claude Mythos Preview였습니다. 최초 프롬프트 이후 사람의 개입 없이, 단 한 번의 스캐폴드 실행 비용 $50 미만으로.

이 글은 Anthropic Frontier Red Team이 2026년 4월 7일 공개한 기술 문서를 바탕으로, Mythos가 이 버그를 **어떻게** 찾아냈는지를 해부합니다.

---

## 1. 세대 교체를 숫자로 보다

Mythos의 등장을 이해하려면 먼저 직전 세대인 Claude Opus 4.6과의 격차를 직시해야 합니다.

Anthropic의 Frontier Red Team은 Opus 4.6을 Firefox 147 JavaScript 엔진 취약점 익스플로잇 작성 과제에 수백 번 투입했습니다. 결과는 성공 **2회**였습니다.

같은 과제를 Mythos Preview로 실행했습니다. 250번의 시도에서 **181회** 작동하는 익스플로잇을 완성했고, 29회 추가로 레지스터 제어(register control)까지 달성했습니다. 한 세대 만에 90배의 성능 향상입니다.

내부 벤치마크에서도 격차는 동일합니다. OSS-Fuzz 코퍼스의 약 7,000개 진입점에 대해 모델별로 실행한 결과, Sonnet 4.6과 Opus 4.6은 1~2티어 크래시를 각각 150~175건 달성했지만 3티어 이상은 각 1건뿐이었습니다. Mythos Preview는 1~2티어 595건, 3~4티어 수건, 그리고 완전한 제어 흐름 탈취(tier 5)를 완전히 패치된 타깃 열 곳에서 달성했습니다.

| 지표 | Opus 4.6 | Mythos Preview | 배율 |
| :--- | :---: | :---: | :---: |
| Firefox 147 익스플로잇 성공 | 2회 | 181회 | **90.5x** |
| CyberGym 취약점 재현 (pass@1) | 66.6% | 83.1% | **+16.5%p** |
| SWE-bench Pro | 53.4% | 77.8% | **+24.4%p** |
| Cybench CTF 35문제 | 미포화 | **100%** | 완전 포화 |
| OSS-Fuzz tier-5 달성 | 1건 | 10건 | **10x** |

Cybench CTF 35문제를 100% 통과하면서 Anthropic의 Red Team은 기존 벤치마크가 더 이상 유의미한 평가 기준이 되지 못한다고 판단했습니다. 그래서 실제 제로데이 취약점 발견으로 평가 기준을 전환했습니다.

결과는 이 글이 설명할 세 개의 사례입니다.

## 2. 스캐폴드: 단순함이 핵심이었다

Mythos의 탐색 방법론을 이해하면 반직관적인 사실을 마주합니다. 구조가 단순합니다.

Anthropic은 인터넷과 다른 시스템으로부터 격리된 컨테이너를 실행하고, 테스트 대상 프로젝트와 소스 코드를 올립니다. 그리고 Claude Code와 Mythos Preview를 실행하며, 한 문단짜리 프롬프트를 줍니다. 본질적으로 "이 프로그램에서 보안 취약점을 찾아달라"는 내용입니다.

이후는 에이전트의 영역입니다.

```
[스캐폴드 구조]

1. 격리 컨테이너 실행
   └─ 소스 코드 + 빌드 환경 포함
   └─ 인터넷 차단, 다른 시스템과 격리

2. 파일 우선순위 채점 (사전 단계)
   └─ Claude가 각 파일을 1~5점으로 채점
   └─ 1점: 취약점 가능성 없음 (상수 정의 등)
   └─ 5점: 인터넷 데이터 파싱, 인증 처리 등
   └─ 점수 높은 파일부터 탐색 시작

3. 병렬 에이전트 실행
   └─ 에이전트별로 다른 파일 담당
   └─ 동일 버그 중복 발견 방지
   └─ 탐색 다양성 극대화

4. 각 에이전트의 탐색 루프
   └─ 코드 읽기 → 취약점 가설 수립
   └─ 실제 실행으로 확인/반증
   └─ 디버거·디버그 로직 추가 반복
   └─ 결과: 없음 or 버그 리포트 + PoC

5. 검증 에이전트 (최종 단계)
   └─ "이 버그 리포트가 실제이고 중요한가?"
   └─ 사소한 버그 필터링
   └─ 고심각도 취약점만 통과
```

각 에이전트는 코드를 읽어 취약점이 있을 수 있다는 가설을 세우고, 실제 프로젝트를 실행해 그 가설을 확인하거나 반증하며, 필요하다면 디버그 로직을 추가하거나 디버거를 사용하고, 마지막에는 버그가 없다는 결론 또는 개념 증명 익스플로잇과 재현 단계가 담긴 버그 리포트를 출력합니다.

이 구조의 핵심은 세 가지입니다. **격리**로 안전성을 확보하고, **우선순위 채점**으로 효율을 높이고, **검증 에이전트**로 노이즈를 제거합니다. 복잡한 특수 목적 도구 없이, Claude Code가 그냥 코드를 읽고 실행하고 생각합니다.

## 3. 사례 1: OpenBSD TCP SACK — 27년짜리 버그

### 배경: SACK이란 무엇인가

TCP의 기본 확인 응답(ACK) 메커니즘에는 구조적 한계가 있습니다. 패킷 1, 2, 4~10을 받고 3번을 놓쳤다면, 수신자는 2번까지만 ACK를 보낼 수 있습니다. 4~10을 받았다는 사실을 전달할 방법이 없습니다.

RFC 2018(1996년 10월)이 이 문제를 SACK(Selective Acknowledgment)으로 해결했습니다. 연속적인 범위만 알리는 대신 "1~2번과 4~10번을 받았다"처럼 비연속 범위를 정확히 전달할 수 있게 됐습니다. OpenBSD는 1998년에 SACK을 구현했습니다.

### 구현 방식: hole 연결 리스트

OpenBSD는 SACK 상태를 **hole의 단방향 연결 리스트**로 추적합니다. Hole은 송신 측이 보냈지만 수신 측이 아직 ACK하지 않은 바이트 범위를 의미합니다.

예를 들어, 송신자가 1~20바이트를 보냈고 수신자가 1~10과 15~20을 ACK했다면, 리스트에는 11~14를 커버하는 단일 hole이 남습니다. 새로운 SACK이 들어올 때마다 커널은 리스트를 순회하며 해당 범위가 커버되는 hole을 줄이거나 삭제합니다.

### 결함 1: 시작 범위 경계 검사 누락

첫 번째 버그는 SACK 범위의 시작값에 대한 경계 검사 누락입니다.

OpenBSD는 SACK 블록의 **끝값**이 현재 송신 윈도우 안에 있는지 검증합니다. 하지만 **시작값**에 대한 검증은 없습니다. 실제로는 무해합니다. -5번 바이트부터 시작하는 ACK를 받아도 0번부터 시작하는 것과 실용적으로 동일합니다. 그래서 아무도 이 검사를 추가하지 않았습니다.

### 결함 2: NULL 포인터 역참조

두 번째 버그는 단일 SACK 블록이 상태 리스트의 유일한 hole을 삭제하면서 동시에 새 hole을 추가하는 경로를 트리거할 때 발생하는 NULL 포인터 역참조입니다.

커널이 hole 리스트를 순회할 때, 해당 hole을 삭제하면서 바로 그 뒤에 새 hole을 append하려 하면 포인터가 NULL이 됩니다. 이미 해제된 유일한 노드를 통해 쓰기를 시도하기 때문입니다.

### 체이닝: 32비트 정수 오버플로우

TCP 시퀀스 번호는 32비트 정수이며 순환합니다. OpenBSD는 `(int)(a - b) < 0` 방식으로 비교합니다. 두 값의 차이가 2^31 미만인 경우에는 올바른 방법입니다. 정상적인 시퀀스 번호는 항상 이 조건을 만족합니다. 하지만 결함 1로 인해, 공격자는 SACK 블록의 시작값을 실제 윈도우에서 약 2^31 거리에 배치할 수 있습니다.

그 거리에서 뺄셈이 두 비교 연산 모두에서 부호 비트를 동시에 오버플로우합니다. 커널은 공격자의 시작값이 hole의 아래이면서 동시에 가장 높은 ACK 바이트 위에 있다고 판단합니다. 불가능한 조건이 도달 가능해집니다. 유일한 hole이 삭제됩니다. append가 실행됩니다. 커널은 NULL 포인터에 씁니다. 머신이 크래시됩니다.

```
공격 흐름:
  1. TCP 연결 열기 (인증 불필요)
  2. SACK 블록 시작값 = 정상_윈도우 + 2^31 설정 (결함 1 이용)
  3. 뺄셈 오버플로우 → 두 비교 동시 만족 (불가능한 상태)
  4. 유일한 hole 삭제 + append 트리거 → NULL 포인터 역참조 (결함 2)
  5. 커널 패닉 → 호스트 다운
```

### 비용과 의미

OpenBSD 스캔 전체 캠페인 비용은 $20,000 미만이었습니다. 이 버그를 실제로 발견한 단일 실행 비용은 $50 미만이었습니다.

OpenBSD는 1998년부터 보안을 최우선으로 설계된 운영체제입니다. 전 세계 방화벽과 중요 인프라에서 실행됩니다. 그 코드베이스는 27년 동안 지속적으로 감사받았습니다. 이 버그는 현재 패치됐습니다.

## 4. 사례 2: FFmpeg H.264 — 500만 번의 퍼저가 놓친 것

### 버그의 역사

FFmpeg는 세계에서 가장 널리 사용되는 멀티미디어 처리 라이브러리입니다. H.264 디코더 결함은 2003년 커밋으로 처음 들어왔고, 2010년 리팩터링을 거치면서 완전히 노출됐습니다. 그 이후 이 코드를 점검한 모든 퍼저와 모든 리뷰어를 16년간 빠져나갔습니다.

자동화 도구는 결코 이것을 잡지 못했습니다. 시도가 부족해서가 아닙니다. 500만 번의 퍼즈 테스트가 있었습니다. 결과는 0건이었습니다. Mythos는 코드를 직접 분석해서 찾아냈습니다.

결함의 정체는 **sentinel collision**입니다. H.264 디코더 내부의 센티널 값이 특정 조건에서 충돌하며 out-of-bounds write가 발생합니다. 이 취약점은 악의적으로 조작된 H.264 비디오 파일 하나로 트리거할 수 있습니다.

### 퍼징이 못 잡은 이유

퍼징은 무작위 또는 변이된 입력을 대량으로 프로그램에 주입해 크래시를 유발하는 방법입니다. 코드 커버리지를 높이는 방식으로 발전해왔고, 현대 소프트웨어 보안의 핵심 도구입니다. 그러나 구조적 한계가 있습니다.

퍼징은 **실행 경로**를 탐색합니다. 특정 코드 경로에 도달하는 입력을 찾는 데 탁월합니다. 하지만 두 개의 독립된 조건이 동시에 성립해야 하는 취약점, 혹은 코드의 의미론적 구조를 이해해야만 발견할 수 있는 취약점에는 약합니다.

FFmpeg H.264 버그는 후자였습니다. 퍼저는 코드를 블랙박스로 실행합니다. Mythos는 코드를 읽고 의미를 파악합니다. 센티널 값이 어떤 역할을 하는지, 언제 충돌이 가능한지를 추론한 것입니다.

| 탐색 방법 | 작동 원리 | 강점 | 한계 |
| :--- | :--- | :--- | :--- |
| **퍼징** | 무작위·변이 입력 대량 주입 | 단순 메모리 오류, 크래시 탐색 | 의미론적 버그, 복잡한 조건 취약 |
| **정적 분석** | 코드 패턴 규칙 기반 스캔 | 알려진 패턴 탐지 빠름 | 알려지지 않은 패턴 탐지 불가 |
| **LLM 코드 추론** | 코드 의미 이해 + 가설 수립 + 실행 검증 | 복잡한 로직 버그, 다중 조건 취약점 | 높은 비용, 할루시네이션 위험 |

Mythos가 메모리 안전 취약점에 집중하는 이유 중 하나가 여기에 있습니다. Address Sanitizer 같은 도구로 결과를 기계적으로 검증할 수 있기 때문입니다. Opus 4.6으로 Firefox 112 버그를 보고했을 때 모든 버그가 진짜 취약점으로 확인됐습니다. LLM의 취약점이 할루시네이션이 아님을 보장하는 체계가 있습니다.

## 5. 사례 3: FreeBSD NFS — 사람 없이 완성된 익스플로잇

### 버그 개요

FreeBSD NFS 서버에서 17년 된 원격 코드 실행 취약점(CVE-2026-4747)이 발견됐습니다. 조건은 가혹합니다. 인증되지 않은 원격 공격자가 NFS 서버에 완전한 root 접근 권한을 획득할 수 있습니다.

### 자율 익스플로잇 작성

Mythos Preview는 FreeBSD의 NFS 서버에 대한 원격 코드 실행 익스플로잇을 자율적으로 작성했습니다. 20개의 가젯으로 구성된 ROP 체인을 여러 패킷에 걸쳐 분할해 인증되지 않은 사용자에게 완전한 root 접근 권한을 부여했습니다.

ROP(Return-Oriented Programming)는 현대 익스플로잇 기법 중 가장 정교한 부류에 속합니다. 실행 가능한 새 코드를 주입하는 대신, 이미 메모리에 있는 코드 조각(가젯)의 끝부분 `ret` 명령어를 연결해 원하는 동작을 구성합니다. NX bit, DEP 같은 현대 메모리 보호 기법을 우회하기 위해 개발됐습니다.

20개의 가젯을 식별하고, 이를 네트워크 패킷에 걸쳐 분할 전송하는 체인을 설계하는 작업을 Mythos가 **초기 프롬프트 이후 어떤 사람의 개입도 없이** 완성했습니다.

보안 교육을 전혀 받지 않은 Anthropic 엔지니어들이 Mythos Preview에 밤새 원격 코드 실행 취약점을 찾아달라고 요청하고 다음 날 아침에 완전히 작동하는 익스플로잇을 받아볼 수 있었습니다.

이 지점이 Mythos를 이전 세대와 구분하는 결정적 차이입니다. Opus 4.6은 취약점을 발견하는 데는 뛰어났지만, 발견한 취약점을 실제 동작하는 익스플로잇으로 전환하는 성공률이 사실상 0%에 가까웠습니다. Mythos는 발견과 익스플로잇을 하나의 자율적 루프로 완성합니다.

---

## 6. 의도치 않은 능력의 등장

이 능력들은 설계되지 않았습니다.

Anthropic은 Mythos Preview가 이 능력들을 갖도록 명시적으로 훈련하지 않았습니다. 오히려 코드, 추론, 자율성의 일반적인 개선에 따른 하류 결과로 나타났습니다. 취약점 패치를 더 효과적으로 만드는 동일한 개선이 취약점 공격도 더 효과적으로 만듭니다.

이것이 Anthropic이 Mythos를 일반에 공개하지 않은 근본 이유입니다. 능력을 제거할 수 없습니다. 방어에 유용한 코드 추론 능력과 공격에 유용한 코드 추론 능력은 동일한 능력입니다.

역사적으로 보안 도구는 방어자에게 더 유리했습니다. AFL 같은 현대 퍼저는 처음 등장했을 때 공격자가 취약점을 더 빠르게 찾을 수 있게 한다는 우려가 있었습니다. 실제로 그랬습니다. 하지만 결국 퍼저는 보안 생태계의 핵심 방어 도구가 됐습니다. OSS-Fuzz는 오픈소스 소프트웨어 보안에 막대한 자원을 투입하고 있습니다.

Anthropic은 같은 일이 결국 여기서도 일어날 것이라고 믿습니다. 보안 지형이 새로운 균형점에 도달하면, 강력한 언어 모델은 공격자보다 방어자에게 더 유리하게 작용할 것입니다. 우위는 이 도구를 가장 잘 활용하는 쪽이 가져갈 것입니다. 단기적으로는 프론티어 랩이 주의를 기울이지 않으면 공격자가 유리할 수 있습니다. 장기적으로는 방어자가 자원을 더 효율적으로 배분하고, 새 코드가 출시되기 전에 이 모델로 버그를 수정할 것이라고 예상합니다.

하지만 그 전환 기간이 문제입니다.

## 7. 취약점 발견 비용의 붕괴

이 사건이 보안 업계에 던지는 가장 중요한 함의는 수치에 있습니다.

27년간 살아남은 OpenBSD 버그를 발견하는 데 든 비용은 $50 미만입니다. 이것은 취약점 발견의 경제학을 근본적으로 바꿉니다.

기존 보안의 암묵적 가정은 "찾기 어렵다 = 공격하기 어렵다"였습니다. 전문 보안 연구자가 수주에서 수개월을 투자해야 발견할 수 있는 취약점은, 공격자에게도 그만큼의 비용이 필요했습니다. 그 비용이 방어선이었습니다.

Mythos급 모델이 제로데이 발견 비용과 시간을 사실상 0에 가깝게 줄이면, 발견과 광범위한 패치 배포 사이의 시간 차이가 위협 행위자에게 매우 위험한 익스플로잇 윈도우를 제공합니다.

패치 사이클이 이 속도를 따라잡을 수 없다는 것이 문제의 핵심입니다. Mythos가 Project Glasswing 첫 달에 발견한 취약점 10,000건 중 99% 이상이 발표 시점에 아직 패치되지 않은 상태였습니다. 책임 있는 공개(coordinated disclosure) 프로세스 자체가 새로운 위협에 적응해야 합니다.

## 8. Project Glasswing: 방어자가 먼저

Anthropic은 이 상황에 대한 응답으로 Project Glasswing을 출범했습니다. AWS, Apple, Broadcom, Cisco, CrowdStrike, Google, JPMorgan Chase, Linux Foundation, Microsoft, NVIDIA, Palo Alto Networks 등 50개 초기 파트너 조직에 Claude Mythos Preview 접근 권한을 부여해 중요 소프트웨어를 보안하는 데 사용하도록 했습니다.

Anthropic은 Project Glasswing을 방어 목적으로 프론티어 모델 능력을 활용하는 "긴급한 시도"라고 표현했습니다. $1억 달러의 Mythos Preview 사용 크레딧과 오픈소스 보안 단체에 대한 $400만 달러의 직접 기부를 약속했습니다.

2026년 6월 기준, 이 파트너십은 15개 이상의 국가에 걸쳐 약 150개 조직으로 확대됐습니다. 전력, 수도, 헬스케어, 통신, 하드웨어 산업의 기관들이 포함됩니다.

동시에 Anthropic은 Mythos급 능력이 수 주 안에 일반 사용자에게도 제공될 것이라고 예고했습니다. 방어자들이 먼저 대비하는 시간을 버는 것이 Glasswing의 핵심 논리입니다.

## 마치며: 새로운 기준선

Mythos가 보여준 것은 AI가 보안 연구에 도움이 된다는 사실이 아닙니다. 그것은 이미 알려진 사실이었습니다. Mythos가 보여준 것은 **기준선이 바뀌었다**는 것입니다.

27년 된 버그, 500만 번의 퍼저를 통과한 버그, 17년간 숨어있던 원격 코드 실행 취약점. 이것들은 예외적으로 어려운 버그였습니다. 그런데 이제 $50짜리 단일 실행이 그것을 찾아냅니다.

이 사실은 두 가지 방향으로 동시에 작용합니다. 방어자는 이전에는 불가능했던 규모의 코드베이스 감사를 수행할 수 있게 됩니다. 공격자도 이전에는 불가능했던 속도로 취약점을 발굴할 수 있게 됩니다.

어느 쪽이 더 먼저, 더 잘 사용하느냐가 앞으로 몇 년의 보안 지형을 결정할 것입니다. Project Glasswing은 그 경쟁에서 방어자를 앞서게 하려는 시도입니다. 결과는 아직 열려 있습니다.
